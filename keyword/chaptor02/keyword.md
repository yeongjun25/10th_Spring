# 1. RESTful API란?

REST 아키텍처 스타일을 따르는 API를 RESTful API라고 부른다. (REST API와 같은 의미)

---

## REST API 6가지 원칙

### 1. 균일한 인터페이스
- API 형태가 일관되어야 한다.
- URI로 리소스를 식별할 수 있어야 한다. (URI는 '명사+복수형'으로 작성)
- HTTP 요청 메시지만 보고도 어떤 요청인지 바로 식별 가능해야 한다.

### 2. 무상태
- 서버는 클라이언트의 이전 요청을 기억하지 않는다.
- 서버는 클라이언트의 모든 요청을 독립적으로 수행한다. (이전 요청 영향 X)

> 💡 예시
> 손님: "아메리카노 한 잔 주세요"
> 직원: "네"
> 손님: "차갑게 주세요"
> 직원: "어떤 거를요??" (이전 주문을 기억하지 않음)

### 3. 계층화 시스템
- 클라이언트는 서버 구조가 어떻게 구성돼 있는지 알 필요가 없다.
- 서버 내부 구조가 어떻든, 클라이언트는 그냥 API 요청만 한다. (코드 수정 필요 X)

### 4. 캐시 가능성
- 서버의 응답 데이터는 캐시 가능 여부를 명시해야 한다.
- 모든 페이지에서 사용되는 데이터 또는 자주 사용되는 데이터는 캐시를 통해 서버 응답 시간과 부하를 줄이면서 성능이 향상될 수 있다.

### 5. 클라이언트-서버 구조
- 클라이언트와 서버를 명확히 구분한다.
- 서로 내부 구현을 몰라도 상관없고, API만 연결되면 된다. (독립적으로 개발 가능)

### 6. 온디맨드 코드 (선택사항)
- 서버는 클라이언트에게 실행 가능한 코드를 전송하여 일시적으로 클라이언트 기능을 확장시키거나 사용자 지정할 수 있다.

---

## REST API 장점 3가지

### 1. 확장성
- 서버가 클라이언트 정보를 기억하지 않기 때문에(무상태) 서버 확장이 쉽다.
- Stateful 방식이라면 서버가 세션을 저장하고 있기 때문에 트래픽이 몰렸을 때 서버 확장이 어렵다. (세션 공유 문제)

### 2. 유연성
- 클라이언트-서버 분리 구조 덕분에 API 연결만 잘 된다면, 내부 코드 구현은 자유롭게 변경될 수 있다.

### 3. 독립성
- API 스펙만 잘 지킨다면, 프론트엔드 ↔ 백엔드 서로 신경 쓰지 않고 독립적으로 개발 가능하다.

# 2. 멱등성이란?
같은 요청을 여러 번 해도 서버의 최종 상태가 동일하다.

---

## HTTP 메서드별 멱등성

### 멱등성 O
1. **GET** : 조회 기능 / 조회만 하니까 서버 상태 변경 없음
2. **DELETE** : 삭제 기능 / 최종 상태 동일
3. **PUT** : 데이터 전체 변경 / 아예 덮어쓰니까 최종 상태 동일

### 멱등성 X
1. **POST** : 데이터 생성 기능 / 데이터가 계속 추가되니까 최종 상태 동일 X

### 멱등성 O or X
1. **PATCH** : 데이터 일부 수정 기능 / 경우에 따라 멱등성 여부가 달라짐

> 예시 1 - 멱등성 O
> ```json
> PATCH /users/1
> { "name" : "제이" }
> ```
> 여러 번 호출해도 name은 여전히 '제이'

> 예시 2 - 멱등성 X
> ```json
> PATCH /users/1
> { "age" : +1 }
> ```
> 호출할 때마다 age 증가. 최종 상태 계속 변경

---
# 3. HTTP 메서드 종류
**GET / POST / PUT / PATCH / DELETE**

1. **GET** : 서버에게 데이터를 요청/조회할 때 사용
2. **POST** : 클라이언트 → 서버 데이터 생성할 때 사용
3. **PUT** : 데이터를 한번에 전체 수정할 때 사용 (덮어 씌우는 개념)
4. **PATCH** : 데이터 일부 수정할 때 사용
5. **DELETE** : 데이터 삭제할 때 사용
