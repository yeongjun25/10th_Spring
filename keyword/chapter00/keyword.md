- PK, FK란?

  [PK]

  **Primary Key의 약자로, 해당 테이블에서 유일하게 행을 식별할 수 있는 컬럼**

    - 중복 불가
    - Null 불가
    - 테이블당 1개만 존재 가능

    ---

  [FK]

  **Foreign Key의 약자로, 다른 테이블의 기본키를 참조하는 컬럼**

    - 테이블 간 관계를 표시
    - 참조 값이 무조건 존재해야 함

    ---

- ERD란?

  [ERD (Entity Relationship Diagram)]

    - 데이터베이스 설계를 시각적으로 표현한 다이어그램 ⇒ 설계도 개념
    - 요구사항 분석 → **ERD 설계** → db 구현(코드 작성)
    - db 구조를 한눈에 파악하기 쉬움 (팀원 소통 good)
- 연관관계란? 그리고 연관관계를 설정하는 방법은?

  **테이블 간에 FK로 연결된 관계 (FK를 가진 테이블이 주인 테이블이 됨)**

  종류: 일대일 관계 / 일대다 관계 / 다대다 관계
    
  ---

    1. **일대일 관계**
    - `@OneToOne (mapped by = “ “)`, `@JoinColumn (name = “ “)` 어노테이션으로 표현
    - FK는 어느 쪽에 두든 상관없지만(일대일 한정), 관례상 ‘조회가 더 많은 쪽’에 둠

    ---

    1. **일대다 관계 (RDS에서 가장 많이 사용하는 관계)**
    - `@OneToMany(mapped by = “ “)`, `@ManyToOne` 어노테이션으로 표현
    - FK는 ‘다’ 테이블이 가짐

    ---

    1. **다대다 관계 (RDS에서 사용 X)**
    - 중간 테이블을 만들어서 1:N, M:1로 분리하여 표현함

    ---

- 정규화란?

  **테이블 내 데이터 중복을 없애고 이상 현상을 방지하기 위해 테이블을 분리하는 과정**
    
  ---

  이상 현상이란?

    1. **삽입 이상 현상**: 테이블의 열에 필요한 값이 누락되어 테이블에 삽입할 수 없는 경우

  (ex. 새로운 과목 “Spring”을 추가하고 싶은데 수강하는 학생이 아직 없는 경우 → PK=student_id=Null 불가능하기 때문에 삽입 불가능)

    1. **삭제 이상 현상**: 데이터를 삭제할 때 의도하지 않은 중요 데이터까지 함께 삭제되는 경우

  (ex. ‘파이썬’을 유일하게 듣는 ‘제이’를 삭제했을 때 ‘파이썬’ 과목 데이터 자체도 함께 삭제됨)

    1. **업데이트 이상 현상**: 데이터를 수정할 때 일부만 수정되어 데이터 불일치 발생

  (ex. 파이썬 과목 담당 교수가 바꼈을 때, 실수로 특정 행의 교수 수정을 깜빡함)
    
  ---

  종류:  제1 정규화/제2 정규화/제 3정규화 . . .

    1. **제1 정규화**: 모든 컬럼은 하나의 값만 가져야 한다.
    2. **제2 정규화**: 모든 컬럼이 PK 전체에 완전 종속 (부분 종속 삭제)

  → PK가 복합키일 때, PK의 일부에만 종속돼 있는 컬럼이 존재하는 경우 ⇒ 테이블 분리하기

    1. **제3 정규화**: 이행 종속 제거

  → PK가 아닌 일반 컬럼이 다른 일반 컬럼을 결정하는 경우 (A → B,  B → C 결정) ⇒ 테이블 분리하기
    
  ---

  **단점** 존재: 테이블이 많아질수록 JOIN이 많아져서 조회 성능이 떨어질 수 있음
    
  ---

- 반 정규화란?

  **db 성능 향상을 위해 의도적으로 정규화를 되돌려 컬럼 중복을 허용하는 것**

  → 정규화가 많이 진행될수록 테이블 개수가 늘어나 JOIN이 늘어남 ⇒ 성능 저하 원인
    
  ---

  [When Use?]

    - 조회가 압도적으로 많을 때
    - 데이터가 매우 많을 때
    - JOIN 성능이 실제로 문제가 될 때

    ---

  [한 줄 정리]

  정규화는 정확성 우선,  반 정규화는 성능 우선

  정규화를 우선적으로 진행하고 추후 성능 문제 발생 시 반 정규화를 고려
    
  ---

- DB에서의 상속 관계 표현은 어떻게 하는가?
    1. 단일 테이블 전략 (Single Table)

       : 모든 자식 클래스를 하나의 테이블에 모두 표현

  `@Inheritance (strategy = InheritanceType.SINGLE_TABLE)`

  `@DiscriminatorColumn (name = " ")`

  `@DiscriminatorValue(" ")`

  → JOIN 없어서 조회 빠름 / 구현 단순 / 컬럼 수가 많아짐(지저분함) / NULL 컬럼 많이 발생 가능
    
  ---

    1. 조인 전략 (Joined Table)

       : 부모 테이블 + 자식 테이블 각각 분리하여 표현

  `@Inheritance (strategy - InheritanceType.JOINED)`

  → 정규화된 구조 / NULL 컬럼 없음 / 깔끔함 / 고유 컬럼이 많을 때 / JOIN 필요 (성능)
    
  ---

    1. 구현 클래스마다 테이블 전략

       : 부모 없이 자식 테이블만 생성

  `@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)`

  → 각 테이블이 모두 독립적 / 부모 타입으로 조회할 때 UNION 발생 (느림) / 실무에서 자주 사용 X
    
  ---

- 인덱스란?

  **데이터를 빨리 찾기 위한 자료구조**
    
  ---

  [인덱스가 없을 때]

  SELECT * FROM members WHERE name = “제이”;

  → 테이블 전체 스캔 (너무 느림)
    
  ---

  [인덱스가 있을 때]

  SELECT * FROM members WHERE name = “제이”;

  → 인덱스에서 ‘제이’ 위치 바로 찾을 수 있음 (빠름)
    
  ---

  [장점]

  SELECT 조회 속도가 빠르다
    
  ---

  [단점]

  INSERT, UPDATE, DELETE 연산이 느리다.

  따라서, 조회 기능이 자주 사용되는 컬럼에 사용하는 것이 좋다.
